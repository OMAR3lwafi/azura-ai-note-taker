Azora AI NoteTaker – Product Requirements Prompt (PRP) Document
1. Product Overview and Vision
Azora AI NoteTaker is a cross-platform intelligent meeting assistant for iOS and Android (built with Flutter) that transcribes meetings in real-time and provides on-the-fly AI-driven insights. It supports both Arabic (RTL interface by default) and English (LTR), aiming to improve meeting productivity by capturing spoken content and instantly generating summaries, action items, and decisions. The app’s vision is to deliver a simple, secure, and smart note-taking experience that feels like a natural extension of any meeting – whether in-person or virtual.
2. Key Features and Requirements (PRD)
2.1 Live Audio Processing and Speech-to-Text
Real-Time Audio Capture: Utilize the device microphone to continuously record audio at 16 kHz sample rate (optimal for speech recognition
medium.com
). Implement noise reduction filters and a Voice Activity Detection (VAD) mechanism to ignore silence and background noise
pub.dev
. This ensures that only meaningful speech is processed, saving on computation and improving accuracy.
Instant Speech Transcription: Stream the audio to a Speech-To-Text (STT) engine and display transcriptions instantly on screen. The app must handle interim (partial) results and final transcripts seamlessly – users should see words appear nearly as they are spoken. Support bilingual transcription (Arabic and English) out of the box, with accurate handling of different scripts and right-to-left text rendering for Arabic.
2.2 AI-Powered Meeting Insights
Real-Time AI Suggestions: As the conversation unfolds, the app’s AI will provide context-aware suggestions and analysis. For example, if a decision is made, the AI might highlight it; if tasks are mentioned, it could list them. The AI should generate on-the-fly meeting summaries, key decisions, and action items without user prompting. These suggestions appear in a side panel or overlay, updating in real time.
Context Understanding: The AI is tuned to understand meeting context, distinguish between different discussion topics, and identify intents (e.g. assigning a task or scheduling a follow-up). It should be capable of basic intent analysis, so that it knows when the team decided something or when a new task is mentioned, and then label it appropriately (e.g. “Decision: ...”, “Action Item: ...”). This contextual awareness makes the suggestions relevant and actionable.
2.3 User Interface and Experience
Minimal, Clutter-Free Design: Azora NoteTaker’s UI is clean and unobtrusive. During a meeting session, the focus is on the live transcript and AI suggestions – presented in a readable font with clear contrast. Controls are minimal (e.g., a start/stop recording button, language switch if needed) to avoid distracting the user.
Bilingual Interface (RTL/LTR): The app must fully support Arabic (RTL) and English (LTR) layouts. All screens, icons, and text alignments should mirror properly based on locale. For Arabic, use RTL ordering and appropriate fonts; ensure that the design (paddings, icons, navigation flow) adapts to RTL. English uses the standard LTR layout. The default app language is Arabic (with RTL UI), but users can switch to English easily.
Primary Session Screen: The core screen during an active meeting displays the live transcript (continuously updating text area) and the AI suggestions (perhaps as chips or a scrolling list of highlights like summary bullet points, detected tasks, etc.). This screen uses a split view or an overlay to let the user glance at AI insights without losing track of the transcription.
2.4 Privacy and Security
User Consent for Audio: On first run (and whenever needed by OS policies), the app must request microphone access explicitly, explaining that it will record audio for transcription. No audio is recorded without user consent.
Data Encryption: All recorded data and AI-generated content are handled with confidentiality. Transcription data sent to any cloud service (for STT or AI) should be encrypted in transit (HTTPS). If any data is stored on-device (e.g. saved transcripts or audio snippets), it should be encrypted at rest or stored in a secure container.
Auto-Delete Option: Provide a user-controlled setting to automatically delete stored meeting data after a certain period (e.g. auto-delete transcripts after 7 days or a user-specified duration). This helps users ensure old meeting notes don’t linger if privacy is a concern.
2.5 Extended Feature Set (Product Extras)
In addition to the core functionality above, Azora AI NoteTaker includes several advanced features to enhance its usefulness:
Content Sharing and Export: Users can export their meeting notes and summaries in multiple formats – PDF, Markdown, or plain text – for easy distribution. A built-in sharing option allows sending the exported notes via email, messaging apps, or other collaboration tools in one tap.
Content Organization: The app helps users keep track of many meetings. Meetings can be categorized by topic, project, or team. Users can add custom tags to notes (e.g., “Marketing”, “Sprint Planning”) for easy filtering. A global search function (with full-text search through transcripts and summaries) enables users to quickly find past discussions by keyword.
Calendar Integration & Reminders: Azora integrates with the device’s calendar to fetch scheduled meetings. It can automatically create a recording session entry for a calendar event, or remind the user to start the NoteTaker when a meeting is about to begin. After meetings, if the AI identified tasks or follow-ups, the app sends notifications/reminders about those tasks (e.g., “Don’t forget to send the proposal – from Monday’s meeting”). It can also send a smart notification before a recurring meeting starts, summarizing previous notes (“Last week you discussed X, Y, Z – ready to continue?”).
Multi-Speaker Mode: In meetings with multiple participants, the app will attempt to identify and distinguish speakers in the transcript. This could be achieved by speaker diarization techniques (voice differentiation) or by allowing the user to tag speakers. The transcript would label segments with Speaker names or “Speaker 1, Speaker 2” etc. Frequent speakers can have voice profiles saved for recognition. Additionally, the app can display talk time statistics – e.g., a simple breakdown of how long each participant spoke, as a visual at the end of the session.
Advanced Editing Tools: After a meeting, users might refine the notes. The app supports basic rich text formatting in the notes editor: headings, bullet points, bold/italic text, and highlights for important items. Users can also insert images or attachments into the notes (e.g., a photo of a whiteboard, or a document relevant to the discussion). A built-in spell checker (for both Arabic and English) underlines typos or offers corrections, so the final notes are polished.
Offline Mode: The app can function without internet by recording audio locally. If the device is offline during a meeting, it will still capture audio (and possibly do on-device transcription if feasible; otherwise, just record). Once connectivity is restored, the recorded data is sent for processing – transcribing the audio and generating the AI summary retroactively. All data is cached securely on the device during offline use. The app also keeps an automatic backup of notes locally (and optionally to cloud when online) to prevent data loss.
Integrations with Productivity Services: To fit into users’ workflows, Azora NoteTaker can integrate with services like Notion, Trello, or Microsoft To-Do. Users can export or sync action items from a meeting into their task manager of choice (e.g., create a Trello card for each action item, or a To-Do task in Microsoft). The app could allow linking a Notion page to a meeting, so that the transcribed notes and summary are automatically sent to a Notion database, for example. It could also generate follow-up reports or summaries that are emailed or posted to a team workspace.
Video Attachment & Analysis: Users can import or attach a link to a video recording (e.g., a Zoom recording or YouTube video) of a meeting (or any informative video). The app will then transcribe and analyze the video content using the same AI pipeline. This means automatically generating text from the video’s audio (using STT) and then producing a summary and key points via AI. The feature supports both Arabic and English content in videos. For example, a user could input a YouTube link of an Arabic lecture, and the app would produce an Arabic transcript and summary highlights. This extends Azora’s utility beyond live meetings to any recorded content.
Note: Features 4–8 are additional enhancements to increase the app’s value. The core development priority is on items 2.1–2.4 (real-time transcription, AI suggestions, UI/UX, privacy). The extended features can be staged in development once the core is functional.

Cloud STT Service: For more robust continuous transcription, leveraging a cloud service like Google Cloud Speech-to-Text via the google_speech package is ideal. AI Suggestion Engine: The AI that generates summaries and action items could be implemented via a cloud-based Large Language Model (LLM) (such as OpenAI GPT-4 or Anthropic Claude) accessed through an API. 3.3 Technical Considerations and Examples
Audio Recording and VAD: We will use a plugin such as flutter_audio_capture or low-level dart:async streams to capture raw audio from the mic. We configure it for mono PCM at 16000 Hz, 16-bit. The data can be fed into a VAD algorithm (we can leverage the vad package which uses Silero models for VAD
pub.dev
). This will emit events like onSpeechStart and onSpeechEnd, which we use to control when to start/stop sending data to STT to save bandwidth and avoid nonsense transcripts during silence.
Streaming to STT: If using Google Cloud STT, we configure a streaming recognizer with the desired language code (for Arabic, e.g., ar-SA or appropriate locale; for English, en-US) and with interimResults: true
vibe-studio.ai
. We then pipe the audio bytes into the recognizer’s input stream. As responses come in (with isFinal flag), we update the UI text. Google’s API can also perform speaker diarization if we request it (we’d specify diarization_speaker_count or enableSpeakerDiarization in the config). This would label the transcript with speaker tags, which we can map to Speaker 1,2,… and later allow the user to name them.
Platform Speech (alternative): If using the speech_to_text plugin for initial version, we call speech.listen() with appropriate locale. That plugin will handle interim results (it has callbacks for partial results). We must monitor the duration – on some platforms continuous listening might auto-stop after a certain time (for example, iOS might stop after ~1 minute offline). A strategy could be to automatically restart listening in the callback when it ends, to mimic continuous listening. The quality for Arabic recognition via the native engine should be evaluated; otherwise, default to cloud for that language if needed.
AI Suggestion Implementation: We define how often or when to call the AI. One approach: collect transcript segments and every N seconds or every M sentences, send an update. The prompt can be engineered to include only the most recent part of the conversation or a running summary to keep it within token limits. For real-time suggestions, latency is a consideration – we might use a smaller model or a fast API to get quick responses (OpenAI’s GPT-3.5-turbo is faster than GPT-4, for instance, and could suffice for short summaries). We’ll also need to parse the AI response and update the UI. If the AI returns a JSON or a structured text (like “Summary: ..., Decisions: ...”), our code will insert those into the suggestion panel UI.

Performance Optimizations: Real-time processing can be heavy:
